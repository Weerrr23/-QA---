# Задание 4b — NoSQL: MongoDB

## Как хранить данные о бронированиях

Я бы использовал смешанный подход. Данные пользователя хранил бы отдельно — в документ бронирования включается только его id. Смысл в том, что пользователь может иметь много бронирований, и если он поменяет email — не придётся обновлять каждое из них, только один документ в коллекции users.

Данные комнаты я бы частично записывал прямо в бронирование — тип и цену за ночь. Это нужно чтобы не делать лишний запрос каждый раз когда нужно посчитать стоимость или показать детали брони. Остальные поля документа: даты заезда и выезда, статус, итоговая стоимость и история изменений статуса.

## Какие индексы создал бы

Первый и главный — по id комнаты и датам. Он нужен для проверки пересечений дат когда кто-то создаёт новое бронирование. Это самый частый и важный запрос — без индекса база будет перебирать все документы подряд.

Второй — по id пользователя и дате создания. Чтобы быстро показать историю броней конкретного человека.

Третий — по статусу и дате заезда. Нужен для поиска «зависших» броней — тех у которых статус confirmed, но дата заезда уже прошла, а статус не сменился на completed. Это типичная вещь которую QA проверяет при тестировании фоновых задач.

## Как проверял бы данные

Через MongoDB Compass смотрел бы на несколько вещей. Во-первых — бронирования где дата выезда раньше даты заезда, таких быть не должно. Во-вторых — документы где статус не равен pending, confirmed или cancelled, то есть попало какое-то мусорное значение. В-третьих — документы с пустыми обязательными полями.

Ещё полезна вкладка Schema в Compass — она показывает сводку по всем полям сразу: какие типы данных встречаются, где есть null. Удобно когда нужно быстро проверить новые данные после миграции или после большого релиза.
